# Улучшение изображений в браузере (яркость / контрастность / цветность)

Система улучшения изображений, полностью работающая в браузере пользователя:
изображение никогда не покидает устройство, вся обработка выполняется
локально в фоновом Web Worker и не блокирует интерфейс.

## Быстрый запуск

Обработка использует Web Worker, поэтому браузер должен получить файлы
по `http://`, а не по `file://` (это ограничение самих браузеров, не
проекта). Нужен любой локальный статический сервер, например:

```bash
cd vk_practice
npx serve .
# либо
python3 -m http.server 8080
```

Откройте адрес, который выведет команда (обычно `http://localhost:3000`
или `http://localhost:8080`), в браузере. Сборка не требуется — это набор
статических файлов (HTML/CSS/JS + готовые сторонние библиотеки в `vendor/`).

## Как пользоваться

1. Выберите файл (JPG, PNG, BMP или HEIC) или перетащите его в область загрузки.
2. Либо оставьте включённым «Автоматический подбор параметров», либо
   снимите галочку и задайте яркость/контрастность/цветность ползунками.
3. Нажмите «Обработать» — прогресс обработки отображается в реальном
   времени, кнопка «Отменить» позволяет прервать задачу.
4. По готовности результат появится в правой панели предпросмотра;
   кнопка «Скачать результат» сохранит обработанное изображение на диск.

## Архитектура

```
index.html            — демо-страница (форма загрузки, ползунки, предпросмотр)
css/style.css          — стили демо-страницы
src/app.js             — связывает элементы интерфейса с API
src/api/image-enhancer-api.js   — публичный API модуля (см. ниже)
src/decode/format-utils.js       — определение формата, конвертация HEIC→PNG
src/worker/enhance.worker.js     — Web Worker: вся обработка изображения
src/model/tone-model-weights.js  — веса обученной модели подбора параметров (см. ниже)
vendor/                — готовые сторонние библиотеки (без сборки/npm install)
  tf-core.min.js, tf-backend-cpu.min.js, seedrandom.min.js — TensorFlow.js
  heic2any.min.js       — декодирование HEIC/HEIF в браузере
samples/                — пул эталонных изображений (для ручной QA), уже
                          содержит стартовый набор из реальных фото Kodak24
tools/train-tone-model/ — offline-обучение модели (dev-инструмент, НЕ часть
                          загружаемого в браузер кода, в лимит 10 МБ не входит)
```

### Публичный API (`ImageEnhancerAPI`)

Соответствует разделу ТЗ «Рекомендуемые API модуля»:

| Метод | Назначение |
|---|---|
| `submitTask(file, options?)` | Метод постановки задачи. Принимает `File`/`Blob` и необязательные параметры `{ brightness, contrast, saturation }` (диапазон `-1..1`, `null`/не указано — автоматический расчёт) и `{ outputFormat, quality }`. Возвращает `taskId` **сразу**, обработка идёт в фоне. |
| `getTaskStatus(taskId)` | Метод получения статуса задачи. Возвращает `{ taskId, status, progress, error }`. |
| `cancelTask(taskId)` | Метод прерывания задачи. Возвращает `true`, если задача была активна и успешно прервана. |
| `getResult(taskId)` | Метод получения готового изображения. Возвращает `Promise<Blob>` (реджектится, если статус не `done`). |

Событие `taskstatuschange` (через `EventTarget`, `api.addEventListener(...)`)
возникает при каждом изменении статуса или прогресса и содержит
`{ taskId, status, progress, error }` — это и есть «Событие изменения
статуса задачи» из ТЗ.

Возможные статусы (`TaskStatus`): `queued → decoding → analyzing →
processing → encoding → done`, либо `error`/`cancelled` в любой момент.

Пример использования API из кода:

```js
import { ImageEnhancerAPI } from './src/api/image-enhancer-api.js';

const api = new ImageEnhancerAPI();

api.addEventListener('taskstatuschange', (e) => {
  console.log(e.detail.taskId, e.detail.status, e.detail.progress);
});

const taskId = api.submitTask(file, { brightness: null, contrast: null, saturation: null });
// ... дождаться status === 'done' через событие ...
const blob = await api.getResult(taskId);
```

## Почему обработка устроена именно так

**Один Worker на задачу.** Каждая задача обрабатывается в собственном
`Worker`, что делает отмену (`cancelTask`) мгновенной и надёжной — просто
`worker.terminate()`, без сложной системы кооперативных флагов отмены
внутри вычислений.

**Двухстадийный пайплайн, как описано в рекомендациях к проекту** («ИИ
подбирает оптимальные параметры коррекции… вспомогательный алгоритм
применяет эти параметры»):

1. **Стадия 1 — ИИ подбирает параметры.** По пикселям изображения
   вычисляются 6 статистик (средняя яркость, СКО яркости как мера
   контраста, среднее отклонение каналов от яркости как мера
   насыщенности, доля клиппинга в тенях/светах, цветовой сдвиг). Эти 6
   чисел прогоняются через небольшую полносвязную нейросеть
   (6 → 12 ReLU → 3 tanh, 123 обучаемых параметра, веса лежат в
   `src/model/tone-model-weights.js`, ~2 КБ), которая и выдаёт
   предложенные яркость/контраст/насыщенность. Это настоящая обученная
   модель: веса получены градиентным спуском (Adam, MSE-лосс), а не
   подобраны вручную — код и данные обучения лежат в
   `tools/train-tone-model/` (dev-инструмент, не часть загружаемого в
   браузер кода).
2. **Стадия 2 — вспомогательный алгоритм применяет параметры.** Прямой
   проход по пикселям (offset для яркости, растяжение относительно 0.5
   для контраста, интерполяция к люме для насыщенности) применяет уже
   готовые коэффициенты — это простая детерминированная операция, не
   часть модели. Реализован обычным циклом по `TypedArray`, а не
   тензорами TensorFlow.js — см. «Почему циклы по TypedArray, а не
   тензоры TF.js» ниже.

**Как обучалась модель.** Готового набора эталонных фото с
экспертной разметкой на момент реализации не было (см. `samples/`),
поэтому обучение сделано самостоятельно размеченным (self-supervised)
способом на 24 реальных фотографиях тестового набора Kodak24:
для каждого реального (хорошо снятого) фото генерируется синтетически
испорченная версия с ИЗВЕСТНЫМИ параметрами деградации (той же формулой,
что использует сама стадия 2), а целью обучения служит коррекция,
возвращающая статистику испорченного фото к статистике оригинала —
то есть модель учится по-настоящему обращать реальные деградации
экспозиции/контраста/насыщенности, а не копирует одну придуманную
константу. Полностью воспроизводимо: `node tools/train-tone-model/train.js`
(24 фото → 960+ обучающих примеров → ~10 с обучения на CPU). При
появлении реального набора эталонных изображений с экспертной оценкой
скрипт можно переобучить на них вместо синтетической разметки.

**Почему TensorFlow.js, а не тяжёлая свёрточная сеть.** Модель
намеренно маленькая (123 параметра, доли КБ) и работает над 6 числами
(статистикой изображения), а не над самими пикселями — этого достаточно
для подбора параметров тона, а свёрточная сеть по пикселям 15 Мп
кратно увеличила бы время обработки и объём кода без выигрыша в
качестве для этой задачи.

**CPU-бэкенд вместо WebGL/WASM (для самой модели).** Внутри Worker WebGL
требует `OffscreenCanvas` с не всегда стабильной поддержкой во всех
браузерах; модель работает всего над 6 числами, поэтому выбор бэкенда
здесь не влияет на скорость и гарантированно работает везде.

**Почему циклы по TypedArray, а не тензоры TF.js (для обработки пикселей).**
Первая версия применяла коррекцию к пикселям единым тензорным графом
TensorFlow.js (CPU-бэкенд). На реальных фото 15+ Мп это давало ~55 с
обработки — почти вдвое больше лимита ТЗ в 30 с, так как CPU-бэкенд
tf.js — чистый JS с отдельным проходом по всем пикселям на каждую
операцию (slice/cast/div/mean/square/sub и т.д.). Стадии 1 и 2 теперь
считают статистику и применяют коррекцию тремя плотными циклами по
`Uint8ClampedArray`/`Float32Array` (см. `analyzeAndEnhance` в
`enhance.worker.js`) — тот же результат на порядок быстрее, укладывается
в среднее время ~5 с. TensorFlow.js остался только для прямого прохода
самой модели (6 → 12 → 3), где его накладные расходы пренебрежимо малы.

**HEIC конвертируется в главном потоке.** Библиотека `heic2any`
(обёртка над `libheif`, скомпилированным в WebAssembly) обращается к
`document.createElement`, поэтому не может быть запущена внутри Worker.
Сама конвертация контейнера — быстрая операция (доли секунды — единицы
секунд для фото с телефона), и жёсткая обработка яркости/контраста/цвета
всё равно выполняется в Worker, поэтому UI не блокируется на сколько-нибудь
заметное время. Остальные форматы (JPG/PNG/BMP) декодируются прямо внутри
Worker через `createImageBitmap` + `OffscreenCanvas`, без обращения к
главному потоку вообще.

## Соответствие требованиям ТЗ

| Требование | Как выполнено |
|---|---|
| Работа во всех современных браузерах | Используются веб-стандарты (Web Worker, `OffscreenCanvas`, `createImageBitmap`), поддерживаемые Chrome/Edge/Firefox/Safari последних версий |
| Суммарный объём кода ≤ 10 МБ | Фактически ≈ 1.82 МБ: собственный код (`index.html`, `css/`, `src/`, включая веса модели `src/model/tone-model-weights.js` ~2 КБ) — 39 КБ; готовые библиотеки в `vendor/` — 1.78 МБ (`tf-core.min.js` 287 КБ + `tf-backend-cpu.min.js` 129 КБ + `seedrandom.min.js` 2 КБ + `heic2any.min.js` 1.29 МБ). Итого в 5+ раз меньше лимита в 10 МБ. (`samples/` и `tools/train-tone-model/` в этот лимит не входят — не загружаются в браузер, см. разделы выше) |
| ИИ подбирает оптимальные параметры коррекции | Обученная модель (`src/model/tone-model-weights.js`, 123 параметра) предсказывает яркость/контраст/насыщенность по статистике изображения; вспомогательный алгоритм (циклы по `TypedArray` в `enhance.worker.js`) применяет их к пикселям — см. раздел «Почему обработка устроена именно так» |
| Обработка изображений до 15 Мп | Обработка пикселей — тремя плотными циклами по `TypedArray`, линейно масштабируется по числу пикселей; модель работает над 6 статистиками, а не над самими пикселями, поэтому не влияет на масштабируемость |
| Максимум 30 с / в среднем 5 с | Проверено вручную на реальном фото 3527×5290 (18.6 Мп) после перехода с тензоров TF.js на циклы по `TypedArray` (см. «Почему циклы по TypedArray, а не тензоры TF.js»); см. также `samples/README.md` для проверки на своих изображениях |
| JPG, PNG, HEIC, BMP | `src/decode/format-utils.js` + `createImageBitmap` (нативная поддержка JPG/PNG/BMP) + `heic2any` (HEIC/HEIF) |
| Асинхронный режим, без блокировки браузера | Вся тяжёлая обработка — в Web Worker; главный поток остаётся отзывчивым |

## Важное ограничение запуска

Открывать `index.html` двойным щелчком (протокол `file://`) **не будет
работать** — браузеры блокируют создание Worker'ов для страниц, открытых
напрямую с диска. Обязательно запускайте через локальный сервер (см.
«Быстрый запуск» выше).
